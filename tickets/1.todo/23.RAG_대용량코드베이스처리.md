# 23. RAG - 대용량 코드베이스 처리

## 핵심 문제

LLM은 컨텍스트 윈도우(한 번에 읽을 수 있는 토큰 수)가 제한돼 있다. 수십만 라인의 코드베이스를 통째로 프롬프트에 넣는 건 불가능하거나 매우 비싸다.

---

## RAG의 기본 아이디어

> **"전체를 외우는 대신, 필요한 부분만 그때그때 찾아서 읽는다"**

```
[사용자 질문]
     ↓
① Retrieve  : 질문과 관련된 코드 조각을 검색
② Augment   : 검색된 코드를 프롬프트에 삽입
③ Generate  : LLM이 그 컨텍스트를 바탕으로 답변
```

---

## 구체적인 동작 방식

### 1단계: 인덱싱 (사전 준비)

코드베이스 전체를 잘게 쪼개서 **벡터 DB**에 저장한다.

```
CodingAgent.java  →  [메서드 단위로 청크 분리]
    ├── chat() 메서드  →  임베딩 벡터 → DB 저장
    ├── buildSystemPromptWithHistory() →  임베딩 벡터 → DB 저장
    └── close() →  임베딩 벡터 → DB 저장
```

**임베딩**: 텍스트(코드)를 수백 차원의 숫자 벡터로 변환. 의미가 비슷한 코드는 벡터 공간에서 가깝게 위치.

### 2단계: 검색 (쿼리 시)

사용자 질문도 동일한 임베딩 모델로 벡터 변환 후, 가장 유사한 코드 청크를 꺼낸다.

```
질문: "대화 이력은 어디에 저장돼?"
  ↓ 임베딩
[0.23, 0.87, -0.12, ...]
  ↓ 유사도 검색 (코사인 유사도)
검색 결과:
  - JsonlConversationHistoryStorage.java (유사도 0.92)
  - ConversationHistoryStorage.java 인터페이스 (유사도 0.88)
  - session.jsonl 관련 코드 (유사도 0.81)
```

### 3단계: 프롬프트 구성

```
System: 당신은 코딩 에이전트입니다.

[검색된 관련 코드]
// JsonlConversationHistoryStorage.java
public void addConversation(...) { ... }

[사용자 질문]
대화 이력은 어디에 저장돼?
```

## 현재 koogAgent와 비교

| 방식 | 현재 (Tool 기반) | RAG |
|------|-----------------|-----|
| 파일 접근 | `ReadFileTool`로 직접 읽기 | 임베딩 검색 후 자동 삽입 |
| 검색 방식 | LLM이 어떤 파일 읽을지 판단 | 벡터 유사도로 자동 선택 |
| 대용량 처리 | 파일 하나씩 순차 탐색 | 수십만 파일도 ms 단위 검색 |
| 정확도 | 파일명/구조를 알아야 유리 | 의미 기반 검색 |

현재 `CodeSearchTool`이 `grep` 기반 키워드 검색을 하는데, RAG는 이것의 **의미론적(semantic) 버전**이라 볼 수 있다.

---

## 실제 구현 흐름

### 인덱싱 파이프라인

```
git clone → 파일 파싱 → 청크 분할 → 임베딩 생성 → 벡터 DB 저장
```

### 청크 분할 전략 (중요)

| 전략 | 평가 |
|------|------|
| 파일 단위 | 너무 큼 |
| 라인 단위 | 너무 작고 맥락 손실 |
| 메서드/클래스 단위 | 코드에 가장 적합 ← 권장 |
| 오버랩 청크 | 경계 맥락 보존 |

### 임베딩 모델

- `OpenAI text-embedding-3-small` - 범용
- `voyage-code-2` - 코드 특화, 정확도 높음

### 벡터 DB 선택지

- **Chroma** - 로컬 임베디드, 프로토타입에 적합
- **pgvector** - PostgreSQL 확장, 기존 DB 활용 가능
- **Qdrant** - 고성능, 프로덕션 적합
- **Pinecone** - 관리형 클라우드 서비스

### 검색 고도화

- **하이브리드 검색**: 벡터 유사도 + BM25(키워드) 병행
- **Re-ranking**: 검색 결과를 LLM이 재정렬
- **파일 구조 메타데이터**: 패키지명, 파일 경로 등 함께 저장

---

## 한계

- **청크 경계 문제**: 메서드가 여러 파일에 걸쳐 있는 경우 맥락 누락
- **임베딩 모델 품질**: 코드 특화 모델이 아니면 의미 검색 정확도 저하
- **인덱스 최신화**: 코드가 바뀔 때마다 재인덱싱 필요
- **외부 라이브러리**: 인덱스에 없는 코드는 검색 불가

---

## 이 프로젝트에 적용한다면

현재 `CodeSearchTool` (grep 키워드 검색)을 보완하는 형태로,  
질문이 들어올 때 관련 코드 청크를 자동으로 시스템 프롬프트에 삽입하는 **`RagSearchTool`** 을 추가하는 방향이 자연스럽다.

```
기존: CodeSearchTool  →  grep으로 키워드 매칭
추가: RagSearchTool   →  임베딩으로 의미 기반 검색
```
