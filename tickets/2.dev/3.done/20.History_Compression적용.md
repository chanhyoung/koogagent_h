# History Compression 적용하기

## 배경

Sliding Window(`getHistory()`가 최근 N개만 반환)만으로는 오래된 대화가 완전히 사라진다.  
"처음에 뭐 읽었지?" 같은 질문에 답하지 못하는 문제가 발생한다.

**해결책**: 오래된 대화를 버리기 전에 LLM이 요약해서 `summary.md`에 저장한다.  
이후 요청엔 **요약 + 최근 N개 원본**을 함께 전달한다.

## 전략: Sliding Window + Summarization 조합

| 방식 | 장점 | 단점 |
|------|------|------|
| Sliding Window | 최근 대화는 원본 유지 | 오래된 건 완전히 사라짐 |
| Summarization | 오래된 것도 핵심 보존 | 세부 손실, 요약 LLM 비용 |
| **조합 (채택)** | 두 장점 모두 | - |

### 조합 규칙

```
전체 메시지 > COMPRESS_THRESHOLD(10)이면:
  요약 대상: 전체 - 최근 KEEP_RECENT(4)개
  유지 대상: 최근 4개 (원본 그대로)
  LLM에 전달: [요약문] + [최근 4개]
```

## 구현 절차 (TDD 순서)

---

### Step 1: 상수 정의 — `JsonlConversationHistoryStorage`

```java
private static final int COMPRESS_THRESHOLD = 10; // 이 이상이면 압축 시작
private static final int KEEP_RECENT = 4;          // 항상 원본 유지할 최근 메시지 수
```

- `COMPRESS_THRESHOLD = 10`: 너무 자주 압축하면 LLM 비용 증가, 너무 늦으면 Context Window 초과
- `KEEP_RECENT = 4`: user+assistant 쌍이므로 최근 2번 대화가 원본으로 유지됨

### Step 4: `loadAllMessages()` 분리 — `JsonlConversationHistoryStorage`

기존 `getHistory()` 내부의 파일 읽기 로직을 private 메서드로 추출.

```java
private List<Message> loadAllMessages() throws IOException { ... }
```

`compressHistory()`에서도 전체 메시지가 필요하므로 중복을 막기 위해 분리.

---

### Step 5: `getHistory()` 수정 — `JsonlConversationHistoryStorage`

```java
@Override
public List<Message> getHistory() throws IOException {
    List<Message> all = loadAllMessages();
    if (all.size() <= KEEP_RECENT) return all;
    return all.subList(all.size() - KEEP_RECENT, all.size());
}
```

**테스트**:
- 4개 이하: 전체 반환
- 5개 이상: 마지막 4개만 반환

### Step 6: `getSummary()` 구현 — `JsonlConversationHistoryStorage`

### Step 7: `compressHistory()` 구현 — `JsonlConversationHistoryStorage` (핵심!)

```java
@Override
public void compressHistory(MultiLLMPromptExecutor executor, LLModel model) throws Exception {
    List<Message> all = loadAllMessages();

    // 1단계: 압축 필요 여부 확인
    if (all.size() <= COMPRESS_THRESHOLD) return;

    // 2단계: 요약 대상 분리 (전체 - 최근 KEEP_RECENT)
    List<Message> toSummarize = all.subList(0, all.size() - KEEP_RECENT);

    // 3단계: 요약 프롬프트 구성 (이전 요약 + 요약 대상 대화)
    StringBuilder conversationText = new StringBuilder();
    String existing = getSummary();
    if (existing != null) {
        conversationText.append("이전 요약: ").append(existing).append("\n\n");
    }
    for (Message msg : toSummarize) {
        if (msg instanceof Message.User u) {
            conversationText.append("User: ").append(u.getContent()).append("\n");
        } else if (msg instanceof Message.Assistant a) {
            conversationText.append("Assistant: ").append(a.getContent()).append("\n");
        }
    }

    // 4단계: LLM에게 요약 요청
    // Koog prompt DSL: system에 대화 내용, user에 요약 명령
    // executor.execute() 로 직접 호출 (AIAgent 불필요)

    // 5단계: 요약 저장
    Files.writeString(summaryFile, summaryText);
}
```

**이전 요약을 포함하는 이유**: 압축이 여러 번 일어날 때 누적 맥락을 유지하기 위함.  
오래된 메시지를 파일에서 삭제하지 않는 이유: 원본 보존으로 디버깅·복구 가능.

### Step 8: `buildSystemPromptWithHistory()` 수정 — `CodingAgent`

// After

시스템 프롬프트 구조 (Recency Bias 고려 — LLM은 뒷부분에 집중):

```

# Previous Conversation Summary
{summary}                    ← 오래된 맥락 (배경 지식)

# Recent Conversation
User: ...
Assistant: ...               ← 최근 맥락 (현재 맥락, 뒤에 위치)
```

## 테스트 시나리오

### 단위 테스트 (`JsonlConversationHistoryStorageTest`)

| 테스트 | 검증 내용 |
|--------|-----------|
| `getSummary_returnsNullWhenNoFile` | summary.md 없으면 null |
| `getSummary_returnsNullWhenBlank` | 빈 파일이면 null |
| `getSummary_returnsContentWhenExists` | 내용 있으면 그대로 반환 |
| `getHistory_returnsLastFourWhenMoreThanFour` | 5개 이상이면 마지막 4개만 반환 |
| `getHistory_returnsAllWhenFourOrLess` | 4개 이하면 전체 반환 |
| `compressHistory_doesNothingWhenTenOrLess` | 10개 이하면 summary.md 미생성 |
| `compressHistory_createsSummaryWhenOverThreshold` | 11개 이상이면 summary.md 생성 |
| `compressHistory_includesPreviousSummary` | 재압축 시 이전 요약 포함 |

### 단위 테스트 (`CodingAgentTest`)

| 테스트 | 검증 내용 |
|--------|-----------|
| `buildSystemPromptWithHistory_noHistoryNoSummary` | 기본 프롬프트만 반환 |
| `buildSystemPromptWithHistory_withSummaryOnly` | 요약 섹션 포함 |
| `buildSystemPromptWithHistory_withHistoryOnly` | 최근 대화 섹션 포함 |
| `buildSystemPromptWithHistory_withBoth` | 요약 → 최근 대화 순서 보장 |

## 비용 효과

| 방식 | 전송 토큰 | 비용/요청 |
|------|-----------|-----------|
| 압축 전 (최근 20개) | ~15,000 | $0.15 |
| 압축 후 (요약+최근 4개) | ~3,000 | $0.03 |

→ **토큰 80% 절약, 비용 80% 절약**  
요약 LLM 호출은 1회 발생 후 재사용되므로 전체적으로 이득.

## 검증 시나리오 (통합)

1. 11번 이상 대화 후 `summary.md` 파일 생성 확인
2. 생성된 요약 내용이 오래된 대화 핵심을 포함하는지 확인
3. "처음에 뭐 읽었지?" 질문 → 요약 기반으로 올바르게 답변
4. 추가 대화 후 재압축 시 이전 요약이 새 요약에 누적되는지 확인
