# 21. History Compression: 메시지 수 기반 → 토큰 수 기반 전환

## 배경

현재 `COMPRESS_THRESHOLD`와 `KEEP_RECENT`는 **메시지 수** 기반으로 동작한다.

```java
private static final int COMPRESS_THRESHOLD = 10;
private static final int KEEP_RECENT = 4;
```

문제: 메시지 1개의 토큰 수가 수백 배 차이날 수 있다.

```
"안녕"          →    10 토큰
파일 전체 읽기  → 5,000 토큰
```

메시지 수가 같아도 실제 LLM에 전송되는 토큰이 크게 달라지므로
비용 예측과 컨텍스트 품질 관리가 부정확하다.

## 목표

토큰 수 기준으로 압축 임계값과 보존 범위를 결정한다.

```java
private static final int TOKEN_COMPRESS_THRESHOLD = 50_000; // 50K 토큰 초과 시 압축
private static final int TOKEN_KEEP_BUDGET        = 10_000; // 최근 10K 토큰 보존
```

## 구현 방향

### 1단계: 토큰 카운터 도입

옵션 A — API 응답 활용 (서버 사이드):
- `addConversation` 시 API 응답의 `usage.input_tokens`를 함께 저장
- JSONL 각 줄에 `tokenCount` 필드 추가

옵션 B — 클라이언트 토크나이저 (로컬 추정):
- [jtokkit](https://github.com/knuddelsgmbh/jtokkit) 등 Java 토크나이저 사용
- `cl100k_base` 인코딩 기준 토큰 수 추정

### 2단계: ConversationHistoryStorage 인터페이스 변경

```java
// 토큰 수 조회 메서드 추가
int estimateTokenCount() throws IOException;
```

### 3단계: compressHistory 로직 변경

```java
// 메시지 수 기반 (현재)
if (cachedMessageCount() <= COMPRESS_THRESHOLD) return;

// 토큰 수 기반 (목표)
if (estimateTokenCount() <= TOKEN_COMPRESS_THRESHOLD) return;
```

`KEEP_RECENT`도 "최근 N개" 대신 "최근 N 토큰 이내" 방식으로 변경한다.

## 권장 프로덕션 기준값

| 항목 | 값 | 근거 |
|---|---|---|
| `TOKEN_COMPRESS_THRESHOLD` | 50,000 | Claude 200K 컨텍스트의 25% |
| `TOKEN_KEEP_BUDGET` | 10,000 | 코딩 에이전트 3~4턴 분량 |
